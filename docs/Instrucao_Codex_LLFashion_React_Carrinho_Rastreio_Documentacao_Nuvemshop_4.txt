Este documento deve ser enviado ao Codex para incrementar a aplicação já criada, sem refazer

Decisão recomendada
Não usar apenas um link puro para status. O link deve abrir uma tela de acompanhamento do
pedido, com status claro, rastreio, pagamento, prazo e opção de falar com atendente. O link
puro continua existindo, mas apenas como URL segura com token para abrir a tela.
Resumo da estratégia
- Usar telefone do WhatsApp apenas como vínculo da cliente no backend.
- Não expor telefone, customer_id, cart_id ou order_id interno na URL.
- Criar cart_token seguro para recuperar o carrinho pelo link.
- Criar status_public_token seguro para consultar status do pedido pelo link.
- Salvar carrinho no backend para permitir que a cliente volte depois pelo mesmo link.
- Aplicar controle de concorrência e idempotência para evitar estoque incorreto e pedido
duplicado.
- Criar uma tela React de acompanhamento do pedido, conforme wireframe parte 3.

Instrução principal para o Codex
Contexto:
A aplicação React Web App mobile-first da LLFashion já existe.
Ela foi criada para substituir o carrinho nativo do catálogo do WhatsApp.
O fluxo atual permite montar carrinho atacadista, validar pedido mínimo de R$ 200, coletar dados, calcular/selecionar
envio e gerar link de pagamento Pix pela Nuvemshop.
Objetivo desta tarefa:
Incrementar o que já existe para suportar:
1. recuperação de carrinho por link seguro;
2. vínculo do carrinho com o telefone do WhatsApp;
3. idempotência para evitar operações duplicadas;
4. controle de concorrência no carrinho e no checkout;
5. página de acompanhamento do pedido e rastreio;
6. consulta de status pelo WhatsApp.
Regra importante:
Não recriar a aplicação do zero.
Não refatorar grandes partes sem necessidade.
Não mudar layout, rotas, contratos de API ou serviços existentes sem justificar.
Primeiro analisar o projeto atual e listar exatamente quais arquivos serão alterados ou criados.
O que não deve ser feito
- Não trocar a estratégia principal do React Web App.
- Não voltar a usar o carrinho nativo do catálogo do WhatsApp.
- Não expor telefone ou IDs internos na URL.
- Não criar um segundo carrinho se já existir carrinho aberto para o mesmo token/cliente.
- Não criar dois pedidos se a cliente clicar duas vezes em finalizar.
- Não confiar no frontend para validação de estoque, subtotal mínimo ou status do pedido.
Fluxo desejado para carrinho recuperável
1. Cliente chama no WhatsApp.
2. Backend identifica o telefone do WhatsApp pelo webhook.
3. Backend cria ou recupera customer pelo telefone normalizado.
4. Backend cria ou recupera carrinho aberto para a cliente.
5. Backend gera link com cart_token seguro:
https://llfashionmoda.com.br/carrinho?token={cart_token}
6. Cliente abre o React Web App pelo link.
7. Frontend chama GET /api/carts/{cartToken}.
8. Backend retorna carrinho aberto, itens, subtotal, mínimo, status e avisos.
9. Cliente adiciona ou ajusta itens.
10. Ao voltar pelo mesmo link, o carrinho é carregado novamente.
11. Se o carrinho expirou, mostrar opção de recalcular ou iniciar novo pedido.
12. Se o carrinho já virou pedido, redirecionar para status do pedido.

Concorrência e idempotência
A aplicação precisa impedir problemas como dois cliques em finalizar, duas abas abertas
alterando o carrinho, estoque mudando durante o checkout ou criação duplicada de pedido na
Nuvemshop.
Regras de concorrência
- Adicionar campo version no carrinho para optimistic locking.
- Ao alterar itens, atualizar somente se a versão atual bater com a versão recebida.
- No checkout, usar lock transacional no carrinho ou lock distribuído por cart_id.
- Antes de gerar link Pix, validar estoque de todos os itens novamente.
- Antes de gerar link Pix, validar pedido mínimo de R$ 200 novamente.
Regras de idempotência
- Criar idempotency_key para operações sensíveis.
- Se a mesma operação for repetida, retornar a mesma resposta anterior.
- Nunca criar dois pedidos para o mesmo carrinho.
- Se checkout_url já existe para o carrinho, retornar o checkout_url existente.
Chaves sugeridas:
ADD_ITEM:{cart_id}:{product_id}:{variant_id}:{quantity}
UPDATE_QUANTITY:{cart_id}:{cart_item_id}:{quantity}
CHECKOUT:{cart_id}
CREATE_DRAFT_ORDER:{cart_id}
GENERATE_PAYMENT_LINK:{cart_id}

Página 6
Tela de acompanhamento do pedido
Criar uma tela React para acompanhamento do pedido. Ela deve abrir pelo link seguro com
status_public_token. O link puro sozinho não é suficiente para uma boa experiência; a cliente
precisa ver informações claras do pedido sem depender de atendimento manual.
Conteúdo da tela
- Número do pedido da Nuvemshop, se já existir.
- Status do pagamento: aguardando Pix, pago, cancelado ou estornado.
- Status operacional: recebido, separando, enviado, entregue.
- Forma de envio selecionada, prazo estimado e valor do frete.
- Código de rastreio e link de rastreio, quando disponível.
- Resumo dos itens ou link para detalhes do pedido.
- Botão para copiar código de rastreio.
- Botão para falar com atendente.
Consulta por WhatsApp
Se a cliente enviar mensagens como:
- status
- meu pedido
- rastreio
- acompanhar pedido
O backend deve buscar pedidos pelo telefone normalizado e responder:
Encontrei seu último pedido
Pedido #1024
Pagamento: Pago
Envio: Em transporte
Rastreio: BR123456789BR
Acompanhe por aqui:
https://llfashionmoda.com.br/pedido/status?token={status_public_token}
Se existir mais de um pedido recente, enviar uma lista resumida para ela escolher.
Webhooks e sincronização
- Acompanhar eventos da Nuvemshop para atualizar pagamento, pedido e envio.
- Ao confirmar pagamento, atualizar status local para PAYMENT_CONFIRMED/PAID.
- Ao pedido entrar em separação, atualizar para SEPARATING_ORDER se a informação existir.
- Ao envio ser criado, salvar shipping_tracking_number e shipping_tracking_url.
- Ao entregar, atualizar status para DELIVERED.
- Se a Nuvemshop não retornar alguma informação, mostrar texto amigável: 'Aguardando
atualização da loja'.

Tela Parte 3 - Acompanhamento do pedido
Sim: além do link público seguro, é recomendável criar uma tela visual para acompanhamento. O link puro deve
abrir esta tela, e não apenas retornar JSON ou redirecionar direto para uma transportadora.
URL pública segura
https://llfashionmoda.com.br/pedido/status?token={status_public_token}
- O token deve ser aleatório, não sequencial, expirar ou poder ser revogado, e não revelar dados internos.
- Não permitir consulta de pedido apenas por número do pedido ou telefone na URL.
- Se quiser reforçar segurança, pedir confirmação dos últimos 4 dígitos do telefone antes de exibir detalhes
completos.
Wireframe funcional da tela de status
- Cabeçalho: LLFashion Moda + número público do pedido.
- Card de status: Aguardando pagamento, Pago, Separando pedido, Enviado, Entregue ou Cancelado.
- Linha do tempo: Pedido criado -> Pagamento -> Separação -> Envio -> Entrega.
- Resumo com miniaturas dos produtos, quantidades, variações e total.
- Pagamento: status Pix e botão para abrir/copiar link quando pendente.
- Envio: forma de envio, prazo, código de rastreio e botão “Acompanhar entrega”.
- Suporte: botão “Falar com atendente no WhatsApp”.
Estados recomendados
PAYMENT_PENDING  -> mostrar botão/link Pix
PAYMENT_PAID     -> “Pagamento confirmado”
ORDER_SEPARATING -> “Estamos separando seu pedido”
SHIPPED          -> código e link de rastreio
DELIVERED        -> “Pedido entregue”
CANCELLED        -> motivo e contato de suporte
ERROR            -> mensagem amigável e contato de atendente
